# v0.2 Real User and Aquarium Management — Design Spec

Date: 2026-05-19
Status: Approved
Milestone: v0.2

## Goal

Make Aquarium Master usable by a real owner managing their own aquariums. Replace demo-token auth with JWT, enforce user data isolation, and add full CRUD UI for aquariums, fish stocks, and devices.

## Decisions

| Decision | Choice |
|---|---|
| Scope | All of v0.2 in one pass |
| Auth | JWT + WeChat mock compatibility |
| Aquarium creation UX | Single-page form |
| Fish species selection | Species library search + custom input |
| Seed data strategy | Demo user with all seed data associated |
| Implementation approach | Vertical slices, feature by feature |

## 1. Auth and Session Management

### Current State

`POST /api/auth/wechat-login` returns `demo-token-${user.id}`. Frontend does not persist tokens. No guard mechanism.

### Backend Changes

- Add `jsonwebtoken` dependency.
- Sign JWT containing `userId` and `openId` on login.
- When WeChat credentials exist: real `jscode2session` flow.
- When WeChat credentials are absent: `mock-openid-${code}` auto-signs a valid JWT. Zero-config local development.
- `login` returns `{ accessToken, user }`.
- Token expiry: 7 days.

### Frontend Changes

- On login success, store token via `Taro.setStorageSync`.
- `services/api.ts` request interceptor attaches `Authorization: Bearer <token>`.
- On app launch, check token validity; clear and reset if expired.
- Profile page shows login status and supports logout.

### Seed Data

- `init-db.sql` creates a demo user with `openId = 'mock-openid-demo'`.
- All seed aquariums, fish stocks, devices, records, and reminders are associated with this demo user.
- Logging in with code `demo` gets the demo user's JWT and sees all seed data.

## 2. User Isolation and API Guards

### Current State

`Aquarium.userId` is nullable. Queries have no user filter. Any request sees all data.

### Database Changes

- Assign all existing null-`userId` aquariums to the demo user.
- Make `Aquarium.userId` required (NOT NULL).
- Related queries (water-quality, maintenance, reminders) are scoped via `Aquarium.userId`.

### API Changes

- `JwtAuthGuard` applied globally (login endpoint excluded via `@Public()` decorator).
- New `@CurrentUser()` param decorator extracts user from `request.user`.
- All aquarium queries: `where: { userId: currentUser.id }`.
- Reminder queries: `where: { userId: currentUser.id }`.
- Water-quality and maintenance queries: scoped through aquarium ownership.
- API returns 401 for missing/invalid tokens.

### Frontend Changes

- Unauthenticated home shows empty state with login prompt.
- After login, load the user's aquarium list.
- On 401 response, redirect to login.

## 3. Aquarium Management

### Current State

Backend CRUD exists but frontend has no create/edit pages. Users can only view seeded data.

### Backend Changes

- Add DTOs: `CreateAquariumDto`, `UpdateAquariumDto`.
- Fields: `name` (required), `volumeLiters`, `lengthCm`, `widthCm`, `heightCm`, `species` (freshwater/planted/marine), `status` (RUNNING/MAINTENANCE/PAUSED), `coverUrl`.
- Create automatically sets `userId` to current user.
- Delete checks for active reminders before allowing removal.

### Frontend Changes

- **Create page** `pages/aquarium-create/index.tsx`: single-page form with name, type selector, dimensions (auto-calculate volume from L x W x H), status selector, cover upload.
- **Edit page** `pages/aquarium-edit/index.tsx`: reuses form component with pre-filled data.
- Home page: floating "+" button navigates to create page.
- Tank detail: edit icon in top-right corner navigates to edit page.
- Delete: bottom of edit page with confirmation dialog.

### Form Layout

1. Basic info: name, type, cover image.
2. Dimensions: length/width/height (cm) with auto-calculated volume (L × W × H ÷ 1000 = liters).
3. Status: running / maintenance / paused.

## 4. Fish Stock Management

### Current State

`AquariumStock` and `FishSpecies` models exist in schema. No management UI or dedicated API endpoints.

### Backend Changes

- New `fish-stock` module (controller, service, DTOs).
- `GET /api/aquariums/:id/fish-stocks` — list fish in an aquarium.
- `POST /api/aquariums/:id/fish-stocks` — add fish. DTO: `speciesId` or `customName` (at least one required), `quantity` (required), `color?`, `note?`.
- `PATCH /api/fish-stocks/:id` — update quantity, note, etc.
- `DELETE /api/fish-stocks/:id` — remove.
- `GET /api/fish-species` — search species library with `?q=` fuzzy search.

### Frontend Changes

- Fish stock list embedded in tank-detail page. Each entry shows species name, quantity, color indicator.
- **Add fish page** `pages/fish-add/index.tsx`:
  - Search box filters `FishSpecies` in real-time (shows Chinese name, scientific name, image).
  - "Custom species" toggle for manual name input.
  - Quantity input, color picker, note field.
- Edit/delete: tap fish card to enter edit mode. Modify quantity or delete.

### Species Search Interaction

- Type keyword to filter `FishSpecies` table via API.
- Selecting a species auto-fills `speciesId` and `displayName`.
- Toggle to "custom input" mode bypasses species library.

## 5. Device Management

### Current State

`AquariumDevice` model exists (type, name, status, powerWatts, flowRateLph, schedule, note). No management UI or dedicated endpoints.

### Backend Changes

- New `devices` module (controller, service, DTOs).
- `GET /api/aquariums/:id/devices` — list devices for an aquarium.
- `POST /api/aquariums/:id/devices` — add device.
- `PATCH /api/devices/:id` — update device info.
- `DELETE /api/devices/:id` — remove device.
- DTO fields: `type` (filter/light/heater/airpump/co2/other), `name` (required), `status`, `powerWatts?`, `flowRateLph?`, `schedule?`, `note?`.

### Frontend Changes

- Device list embedded in tank-detail page, grouped by type (filters, lights, heaters, air pumps, CO2, other).
- **Add device page** `pages/device-add/index.tsx`:
  - Device type selector (presets + custom).
  - Name, power (watts), flow rate (LPH, shown conditionally by type), schedule, note.
- Edit/delete: tap device card to enter edit mode.
- Status indicator: green (running), yellow (maintenance), gray (stopped).

## 6. Execution Order (Vertical Slices)

### Slice 1 — Auth Foundation

- Backend: JWT signing, `JwtAuthGuard`, `@CurrentUser()` decorator, login returns token.
- Frontend: token persistence, request interceptor, login/logout flow.
- Seed: create demo user, associate all seed data.
- Verify: login → get token → request with token succeeds → request without token returns 401.

### Slice 2 — User Isolation

- Backend: global guard, userId filtering on all queries, `Aquarium.userId` becomes required.
- Frontend: 401 redirect, home loads user's aquariums, unauthenticated empty state.
- Verify: demo user sees all seed data; new user sees empty list after login.

### Slice 3 — Aquarium CRUD

- Backend: DTO validation, create/update/delete with user ownership check.
- Frontend: create page, edit page, Home "+" button, tank-detail edit entry.
- Verify: create aquarium → refresh shows it → edit name → delete with confirmation.

### Slice 4 — Fish Stock Management

- Backend: fish-stock module, species search endpoint.
- Frontend: tank-detail fish list, add page (species search + custom), edit/delete.
- Verify: add fish from species library → update quantity → delete.

### Slice 5 — Device Management

- Backend: devices module, type presets.
- Frontend: tank-detail device list (grouped), add page, edit/delete, status indicator.
- Verify: add filter → set power and flow → toggle status → delete.

## Acceptance Criteria

- A logged-in user only sees their own aquariums and records.
- A user can create a new aquarium without editing seed SQL.
- Fish and device edits persist through the API and survive reload.
- `corepack pnpm typecheck` passes.
- `corepack pnpm lint` passes.
- `corepack pnpm build:h5` produces a working H5 build where all new features are operable.
- Demo user login shows all seed data; new user starts from a clean state.

## Out of Scope

- Water-quality trend charts (v0.3).
- Reminder repeat-rule structure (v0.4).
- AI diagnosis endpoint (v0.5).
- Species library standalone pages (v0.6).
- Production logging and health checks (v0.7).
- Maintenance record image upload (deferred from original v0.2 to keep scope manageable).
